# DNF手游增幅概率预测工具 - 设计文档

## 项目概述

一个基于 Web 的地下城与勇士手游（DNF Mobile）增幅概率预测工具。用户使用相同成功概率的徽章进行10-20次"垫手"，系统通过马尔可夫链、隐马尔可夫模型（HMM）和贝叶斯更新算法分析垫手序列，预测下一次增幅的成功概率并给出操作建议。用户反馈实际增幅结果后，系统自动校准模型权重，越用越准。

## 目标用户

DNF手游玩家，希望通过科学的垫手数据分析来辅助增幅决策。

## 核心功能

1. **垫手记录**：记录10-20次徽章合成的成功/失败结果
2. **增幅等级选择**：选择当前要增幅的目标等级（+4 到 +20）
3. **徽章等级选择**：选择垫手用的徽章等级（5级到10级），系统自动匹配成功率
4. **概率预测**：基于三个算法模型的加权结果预测下一次成功率
5. **操作建议**：给出具体预测概率百分比和操作建议文字
6. **趋势图表**：展示垫手过程中预测概率的变化曲线
7. **结果反馈**：用户反馈实际增幅结果（成功/失败），系统记录并自动校准模型权重
8. **历史记录**：展示过往所有预测和实际结果的对照，显示系统准确率
9. **数据导入/导出**：支持导出 JSON 文件，换设备后可导入恢复历史

## 算法设计

### 三层算法架构

#### 第一层：马尔可夫链模型

将垫手序列建模为马尔可夫链：

- **状态定义**：基于最近2次结果组合为4个状态（SS, SF, FS, FF）
- **转移矩阵**：根据用户输入的垫手序列构建状态转移概率矩阵
- **预测**：基于当前状态和转移矩阵，预测下一次成功的概率

```
状态转移图:
  SS ──→ SS (连续成功后再成功)
  SS ──→ SF (连续成功后失败)
  SF ──→ FS (交替)
  SF ──→ FF (连续失败开始)
  ...
```

#### 第二层：隐马尔可夫模型（HMM）

假设游戏内部存在隐藏的"运气状态"：

- **隐藏状态（3个）**：
  - H0 = 好运状态（成功概率高于基础值）
  - H1 = 普通状态（成功概率约等于基础值）
  - H2 = 差运状态（成功概率低于基础值）
- **观测值**：每次垫手的成功（1）或失败（0）
- **发射概率**：每个隐藏状态对应不同的成功概率
- **训练**：Baum-Welch 算法，用垫手序列作为观测数据训练 HMM 参数
- **推断**：Viterbi 算法计算当前最可能的隐藏状态
- **预测**：基于推断的隐藏状态，修正目标增幅等级的成功概率

#### 第三层：贝叶斯更新

将基础成功率作为先验概率，垫手结果作为观测证据：

- **先验分布**：Beta(α, β)，其中 α/(α+β) = 基础成功率
- **似然函数**：二项分布，每次垫手的成功/失败
- **后验分布**：每输入一次垫手结果，更新 Beta 分布参数
- **预测**：后验分布的均值作为修正后的成功概率

#### 综合预测

冷启动阶段（<50条记录），三个模型的加权融合：
```
最终概率 = w1 × 马尔可夫概率 + w2 × HMM概率 + w3 × 贝叶斯概率

默认权重: w1=0.3, w2=0.4, w3=0.3
```

成熟阶段（50+条记录），加入经验模型后四个模型融合：
```
最终概率 = w1 × 马尔可夫概率 + w2 × HMM概率 + w3 × 贝叶斯概率 + w4 × 经验概率

默认权重: w1=0.2, w2=0.3, w3=0.2, w4=0.3
（经验模型和 HMM 权重最高，因为它们基于真实数据）
```

经验模型的数据不足时（该模式出现<3次），自动降级为三模型融合。

**置信度**：基于垫手次数和三个模型结果的一致性计算：
- 垫手次数少于5次：置信度很低（< 0.3）
- 垫手5-10次：置信度中等（0.3-0.6）
- 垫手10-20次：置信度较高（0.6-0.85）
- 三个模型结果分歧大时降低置信度

### 操作建议逻辑

基于最终预测概率和基础概率的对比，给出具体概率百分比供玩家判断：

| 条件 | 建议 | 显示内容示例 |
|------|------|-------------|
| 预测概率 > 基础概率 × 1.3 且 置信度 > 0.5 | 建议增幅 | "预测成功率 72.3%，高于基础 50%，建议直接增幅" |
| 基础概率 × 1.1 ≤ 预测概率 ≤ 基础概率 × 1.3 | 可增幅但建议再垫 | "预测成功率 58.1%，略高于基础 50%，可以再垫几手" |
| 基础概率 × 0.8 ≤ 预测概率 < 基础概率 × 1.1 | 建议继续垫手 | "预测成功率 47.5%，接近基础 50%，建议继续垫手" |
| 预测概率 < 基础概率 × 0.8 | 不建议操作 | "预测成功率 35.2%，低于基础 50%，不建议此时增幅" |

**概率展示方式**：
- 仪表盘中央大字显示预测概率百分比（如 `72.3%`）
- 下方对比条：`基础 ████████░░ 50% → 预测 ██████████████░░ 72.3%`
- 建议文字带颜色标识：绿色=建议增幅，黄色=可以再垫，橙色=继续垫手，红色=不建议

### 反馈学习机制（越用越准）

#### 核心循环

```
垫手 → 预测 → 建议 → 用户操作增幅 → 反馈结果(成功/失败)
         ↑                                       │
         └─── 模型校准 + 模式学习 + 参数更新 ←─────┘
```

系统有 **两层学习**：短期校准（每次反馈后调整）+ 长期学习（积累数据后挖掘规律）。

#### 第一层：短期校准（每次反馈后立即生效）

1. **结果记录**：用户点击"增幅成功"或"增幅失败"按钮反馈实际结果
2. **记录内容**：每次反馈保存为一条记录
   ```json
   {
     "timestamp": "2026-04-10T15:30:00",
     "enhance_level": "+10",
     "badge_level": 7,
     "pad_results": ["S","F","F","S","F","F","F","S","F","F"],
     "predicted_rate": 0.72,
     "base_rate": 0.50,
     "recommendation": "enhance",
     "actual_result": "success",
     "model_weights": {"w1": 0.3, "w2": 0.4, "w3": 0.3}
   }
   ```
3. **模型权重自动校准**：
   - 每次反馈后，回溯三个模型各自的预测值与实际结果的偏差
   - 马尔可夫预测72%实际成功 → 马尔可夫得分 +1
   - HMM预测80%实际失败 → HMM得分 -1（高概率预测失败，扣分）
   - 贝叶斯预测70%实际成功 → 贝叶斯得分 +1
   - 统计各模型最近20次的加权准确率，按准确率比例重新分配权重
   - 最小权重不低于0.1，保证不完全依赖单一模型

#### 第二层：长期学习（积累10+条记录后生效）

随着历史数据积累，系统逐步从"依赖默认模型"过渡到"依赖你的真实数据"。

1. **经验模式表**（Empirical Pattern Table）：
   - 统计历史中每种垫手模式对应的实际成功率
   - 按最后N次垫手结果（尾缀模式）分组统计
   - 示例：
   ```
   模式: "连续3次失败后" → 历史中出现15次，实际成功9次 → 经验概率 60%
   模式: "交替SF后"     → 历史中出现8次，实际成功3次  → 经验概率 37.5%
   模式: "连续2次成功后" → 历史中出现12次，实际成功4次 → 经验概率 33%
   ```
   - 随着数据增多，经验模式表覆盖的模式越来越丰富
   - 当历史数据足够（50+条）时，新增第四个模型"经验模型"，权重逐步提升

2. **HMM 在线学习**：
   - 每次反馈后，将新的垫手序列+结果追加到 HMM 训练数据池
   - 定期（每积累5条新数据）用全量历史数据重新训练 HMM 参数
   - 初始 HMM 参数来自默认设定，随数据积累逐步过渡到用户专属参数
   - 训练在服务端完成，训练后返回新参数给前端缓存

3. **马尔可夫链动态扩展**：
   - 初始用2阶马尔可夫链（看前2次）
   - 积累30+条数据后，自动测试3阶马尔可夫链（看前3次）是否更准
   - 如果3阶的交叉验证准确率更高，自动升级到3阶
   - 以此类推，最高到4阶（看前4次结果）

4. **贝叶斯先验更新**：
   - 初始先验 = 社区基础概率
   - 积累用户自己的数据后，用用户历史实际成功率作为先验
   - 如果用户在+10的实际成功率是45%而非官方的50%，先验自动偏向45%
   - 这让系统适应用户的"个人运气"或"特定服务器规律"

#### 学习阶段展示

系统根据历史数据量显示当前"学习阶段"，让用户知道系统有多可靠：

| 阶段 | 条件 | 说明 | 显示 |
|------|------|------|------|
| 冷启动 | 0-4条记录 | 纯算法模型，无个人数据 | "学习中 🟡" |
| 初学 | 5-19条记录 | 开始校准权重，经验表初步建立 | "成长中 🟢" |
| 成熟 | 20-49条记录 | HMM重训练，3阶马尔可夫可能激活 | "较成熟 ✅" |
| 专家 | 50+条记录 | 经验模型参与预测，全面个性化 | "高度个性化 🏆" |

#### 准确率展示

- 历史总准确率："系统预测准确率：68%（20次中14次正确）"
- 按等级分组："+10 增幅预测准确率：75%（8次中6次）"
- 最近10次趋势："近期准确率上升 ↑"
- 学习阶段："当前阶段：成长中 🟢（15条记录）"
/
#### 数据存储

- **主存储**：浏览器 localStorage，键名 `dnf_enhance_history`
- **导出**：点击"导出数据"按钮，下载 JSON 文件包含所有历史记录、校准后的模型权重、经验模式表
- **导入**：换设备后点击"导入数据"，选择之前导出的 JSON 文件恢复全部历史和学习成果
- **数据上限**：localStorage 保留最近200条记录，超出后提示用户导出归档

#### 移动端（<768px）

纵向堆叠布局，所有区块从上到下排列：
1. 增幅设置（折叠式）
2. 垫手记录（全宽，大按钮）
3. 概率仪表盘（居中显示）
4. 模型详情（折叠式）
5. 趋势图（全宽）

### 视觉风格

- **主题**：暗色背景（#1a1a2e）+ 金色强调（#ffd700）+ 红色警告（#ff4444）
- **按钮**：成功按钮金色发光，失败按钮暗红色
- **仪表盘**：弧形进度条，颜色从红→黄→绿表示概率低→高
- **动画**：概率变化时有平滑过渡动画
- **触屏**：所有可交互元素最小 48px 点击区域

## 部署配置

### 前端 (Vercel)

```json
// vercel.json
{
  "buildCommand": "npm run build",
  "outputDirectory": "dist",
  "framework": "vue"
}
```

部署后永久地址：`https://xxx.vercel.app`

### 后端 (Render)

```yaml
# render.yaml
services:
  - type: web
    name: dnf-enhance-api
    runtime: python
    buildCommand: pip install -r requirements.txt
    startCommand: uvicorn app.main:app --host 0.0.0.0 --port $PORT
    plan: free
```

部署后永久地址：`https://xxx.onrender.com`

### 冷启动缓解

- 前端检测 API 请求超时或慢响应时，显示"正在唤醒服务器，请稍候..."
- 可选：配置 UptimeRobot（免费）每5分钟 ping 一次后端保持活跃

## 浏览器兼容性

- Chrome 90+, Safari 14+, Firefox 90+, Edge 90+
- 移动端：iOS Safari 14+, Android Chrome 90+
- 响应式断点：768px（平板/手机分界）

## 非功能需求

- 前端首次加载 < 3秒（Vercel CDN）
- API 响应 < 500ms（热启动时）
- 无需登录，无需服务端数据库
- 历史数据和模型权重存储在浏览器 localStorage
- 支持 JSON 文件导入/导出，可跨设备迁移
- 所有实时计算在服务端完成，校准计算在前端 + 服务端协同
